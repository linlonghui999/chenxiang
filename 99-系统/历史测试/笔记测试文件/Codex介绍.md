# Codex 介绍

## 什么是 Codex

Codex 是 OpenAI 提供的软件开发智能体。它不仅能回答编程问题，还可以在获得授权后读取代码仓库、编辑文件、运行命令、执行测试并检查结果，帮助开发者完成从理解需求到验证交付的完整开发流程。

Codex 适合处理以下工作：

- 阅读和解释陌生代码库
- 实现功能、修复缺陷和补充测试
- 排查构建失败、运行错误和性能问题
- 审查代码并指出潜在风险
- 更新文档、配置和自动化脚本
- 执行重复性的仓库维护任务

Codex 的价值不只是“生成代码”，而是能够结合仓库上下文持续推进任务，并使用实际的构建、测试和运行结果验证修改。

## 主要使用方式

Codex 提供多种工作入口，可以根据场景选择：

| 使用方式 | 适合场景 | 特点 |
| --- | --- | --- |
| 桌面应用 | 管理项目、长任务和多个并行工作 | 提供文件预览、差异检查和任务管理界面 |
| Codex CLI | 在终端中直接处理本地仓库 | 可读取文件、修改代码、运行命令并自动化工作 |
| IDE 扩展 | 在编辑器中边写边改 | 能利用当前打开的文件和选中的代码作为上下文 |
| 云端任务 | 将耗时任务交给托管环境执行 | 适合并行或不需要持续盯守的工作 |

官方说明：[Codex](https://developers.openai.com/codex)、[Codex CLI](https://developers.openai.com/codex/cli)、[Codex IDE 扩展](https://developers.openai.com/codex/ide)。

## 一个典型的工作流程

### 1. 给出清晰目标

先说明最终要达到的结果、相关文件和不能改变的行为。例如：

```text
修复登录页面在网络超时时一直加载的问题，保留现有界面，并为超时场景增加测试。
```

### 2. 让 Codex 检查仓库

Codex 会查看目录结构、现有实现、项目规范和测试方式，再决定修改位置。对于复杂任务，可以要求它先说明发现的问题和实施步骤。

### 3. 实施并验证

Codex 可以编辑代码、运行格式检查和测试，并根据输出继续修正。有效的任务描述应明确验收条件，例如“所有原有测试通过，并新增一个覆盖该缺陷的测试”。

### 4. 审查最终变化

完成后应查看修改摘要、文件差异和测试结果。涉及生产配置、数据删除、权限调整或外部发布时，仍应由人工进行最终确认。

## 如何写出更有效的任务描述

一个实用的任务通常包含四部分：

1. **目标**：希望最终得到什么结果。
2. **范围**：涉及哪些目录、模块或页面。
3. **约束**：哪些行为、接口或样式不能改变。
4. **验证**：用什么测试或现象证明任务完成。

示例：

```text
在用户设置页增加深色模式开关，只修改前端，不改变现有接口。
状态需要持久化到本地存储，并补充组件测试。
完成后运行项目现有的检查和测试命令。
```

## 使用 AGENTS.md 固化仓库规则

仓库可以使用 `AGENTS.md` 告诉 Codex 长期遵守的项目规范，例如：

- 项目的构建、检查和测试命令
- 目录结构与代码风格
- 禁止修改的文件或兼容性要求
- 代码审查和交付标准

`AGENTS.md` 应保持简洁，写入每次处理该仓库都需要遵守的规则。只对单次任务有效的要求，应直接写在任务描述中。更靠近目标文件的 `AGENTS.md` 可以为对应子目录提供更具体的说明。

官方说明：[使用 AGENTS.md 配置自定义指令](https://developers.openai.com/codex/agent-configuration/agents-md)。

## 扩展能力

Codex 还可以通过以下机制扩展：

- **Skills**：封装可重复使用的专业流程、参考资料和脚本。
- **Plugins**：分发由技能、工具和连接能力组成的完整功能包。
- **MCP**：连接文档、代码托管、设计工具或企业系统等外部服务。
- **Subagents**：把大型任务拆给多个专门智能体并行处理。
- **Automations**：定时执行检查、提醒或仓库维护工作。

应从最小且合适的扩展方式开始：仓库规范使用 `AGENTS.md`，可重复流程使用 Skill，需要外部系统时再接入 MCP。

## 权限与安全

Codex 是否能够修改文件、访问网络或运行命令，取决于当前环境的沙箱和审批设置。使用时建议遵循以下原则：

- 只授予完成当前任务所需的权限。
- 执行删除、覆盖、发布和数据迁移前检查目标范围。
- 不要把密钥、令牌或密码直接写入仓库。
- 对外部下载内容和第三方脚本进行来源核验。
- 以实际测试、构建结果和文件差异作为完成依据。

Codex 可以显著提高开发效率，但重要修改仍需要清晰的验收标准和必要的人工复核。

## 快速开始

进入项目目录后，可以在终端启动 Codex CLI：

```bash
codex
```

然后直接描述任务，例如：

```text
请先阅读这个仓库，告诉我如何启动项目，并检查当前测试是否通过。
```

建议从只读的理解和检查任务开始，确认 Codex 已正确识别项目结构后，再逐步交给它实现、测试和审查任务。
