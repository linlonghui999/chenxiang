# 理解 AHA：两个 Agent，如何一起办事？

## 信息摘要

OPPO与阶跃相继联手支付宝，让手机智能体小布和STEPX Neo跨端调用阿宝服务。当A2A协议在身份、资金、责任上卡壳，端侧互联方案另辟蹊径：本地进程通信、各认各账、用户始终在决策节点。这可能是Agent落地的深水区答案。 前两天，OPPO 刘作虎发了条微博，官宣和支付宝的合作。OPPO 手机上的智能体小布，能委托支付宝段智能体阿宝跨端干活。首期落地两百项服务，官方的说法是「一句话办成一件事」 今天下午的 WAIC 现场，阶跃也官宣了同款合作。它的终端 STEPX Neo，也把人下的任务，外包给支付宝的阿宝去办，我还去现场实拍了 两个新闻，也可以当一个行业问题的答案看： Agent 之间除了对话外，还能怎么共事？ 我们的来时路 回望过去的 3 年多的时间，我们对于 AI 的感知，已经悄然发生了变化，从最开始的「聊天机器人」，到「干活的工具」，到现在的「离不开的好大哥」，恍惚间都不知道怎么过来的了 让我们先把视角拉回 2022 年 11 月，那还是个经典的香草时代，没有 ChatBot，没有 Agent，一切都是那么的古朴与祥和 这份宁静，随后被 ChatGPT 的上线打破了。我们从聊天开始，不断把各种任务丢过去。让他写周报，让他做检讨，让他…让他啥都干… 但一个没有手的聪明脑袋，能做的事确实有限，于是各家就八仙过海了 → 23 年 3 月，OpenAI 给 ChatGPT 上了 plugin，年底又推出 GPTs 的 Action → 24 年 11 月，Anthropic 发布 MCP，教 Agent 怎么调工具。去年 12 月，MCP 捐进 Linux 基金会旗下的 AAIF，彻底公共化 这些东西，聚焦的都是同一件事：Agent 怎么访问外界 除此之外，我们也在探索如何让 Agent 之间学会打交道，在这一块最早拿出成绩的是是谷歌，在 25 年 4 月拿出了 A2A 方案，首发拉了 50 多家组织，两个月后捐给 Linux 基金会 那阵子，行业普遍觉得，智能体互联的标准算是定了，但现在一年过去了，A2A 虽也没凉，但也没怎么落地。本最该被它改变的供应链、保险、IT 运维，还全是企业内部的工作流，嘛也没变 为啥呢？ 细吧啦下来，判断一个 Agent 互联方案，只消问三个问题：身份怎么算，钱怎么算，责任怎么算 → 身份：没人想把自家用户，随随便便交给别人 → 钱：A2A 是个开放协议，压根没在支付这事儿上做定义 → 责任：Agent 闯了祸谁背锅，分歧很大 对于协议来说，能握手就能成；但对于商业来说，就得拿它完成算账、收钱、担责，完成闭环才行 两种信任模式 我们现在的 AI 行业，出现了一个巨大 Gap：极聪慧的智能，极差的落地，所以就有了很强的供需诉求 需求这头：手机助手、车机、眼镜、大模型 Harness。这些新设备、新应用，都需要深度的数据服务，需要生活功能 供给那头：过去 20 年，支付宝攒下了颇为可观的家底。出行、政务、医疗、缴费，外加 400 多万小程序啥啥都有，为了把沉淀的服务能力接进阿宝，支付宝还搭了 AI 开放平台，扶持商家 AI 改造，帮阿宝扩服务、扩生态 那么，如何把这些服务对给终端，就成了一个问题。 前两天，我和阿宝的工程师聊了老一会儿，也去看看他们怎么思考的。最初的时候，阿宝也想学 A2A，让两个 Agent 在云端互联。结果卡死在三个地方 → 身份信息无法确立，仅靠硬件信息，是分不清同一台手机上的多个账号的 → 也无法通过内嵌 SDK 解决登录状态，厂商对这件事儿极为排斥 → 防蒸馏无解，只要反复调用，再加上一点小的技巧，就能反推你的服务规则和用户画像 既然这条路堵死，他们换了第二条「端侧互联」： 同一台手机上，两个 App 的 Agent 直接进程通信，不经过任何服务端转发 换句话说，就是跳出云端 A2A 互联的思路，不再尝试不断修补身份、资金、全责的矛盾，而是在从终端架构下手，去设计多智能体协作的落地模式 在这种思路下，不去解决那三个问题，而是让那三个问题不存在 → 身份怎么算？不算。不搞账号互通，各认各的登录态。小布认 OPPO 账号，阿宝认支付宝账号，服务的调度和执行发起都由本地支付宝阿宝发起 → 钱怎么算？不算。支付留在支付宝 App 里，协议层面不碰钱 → 责任怎么算？留给人。授权、下单、支付，三个节点，永远用户本人点头 方案的背后 这个方案里，最核心的是阿宝。设备终端会调用的是阿宝背后一整套 Agent 服务网络，这些服务都被做成一个个面向 Agent 的能力，等着被发现&需要时使用…. 这里要说一下：小布调的是阿宝，跟原来那个支付宝 App，是两码事。对终端厂商来说，接一次阿宝，就接进了整张网络。几百项服务一站式对接，把终端 AI 想要的东西做了个一站式采购 中间管理信息交互的这层，是 AHA，Agent Hub Access。外部智能体想要什么、怎么调用，由这个 Hub 统一把关 在执行任务的过程中，其任务分配时在本机两个 App 之间直连联系，不走云端。至于结果，则用卡片渲染，传回对话流，脱敏展示，原始数据不出支付宝 你会问？厂商为什么愿意接入，支付宝为什么愿意开开放，其实也是各取所需。终端 AI 要兑现帮助人解决问题，绕不开真实服务和支付；而 Agent 希望服务更多的人群，也绕不开各种物理入口 具体来说 除了刚才阶跃的 Amoo 之外，让我再举一个例子，比如你对手机说一句： 帮我查下这个月电费，欠了就直接充 200，再顺手收一下蚂蚁森林的能量 话音落下，两个 Agent 开始接力¹ 1 · 理解和规划 小布先拆需求，缴费、收能量，两件事。结合你的位置和日程，排出方案。这一步，阿宝并不参与 2 · 端侧握手 小布用手机本地的进程间通信，唤起支付宝，把任务递给阿宝 3 · 服务发现和路由 阿宝这边，缴费、公益各有自己的子智能体，任务&鉴权被路由过去 4 · 执行分档 简单的事，直接办。要你补充信息的，会在流式对话里结合图形化界面引导用户提供金额，然后再交给阿宝自动执行 5 · 确认和支付 授权、下单、支付，留给用户本人 6 · 渲染和回传 结果以卡片的形式，回到小布的对话流。履约状态异步回传，任务在后台继续走 ¹ 该能力已在 ColorOS 16 机型全量上线，需小布助手 12.8.4、支付宝 12.12.1 及以上版本 两个案例，两件事 上面有两个案例：OPPO 和阶跃，对应的其实是两件事 从 OPPO 的实践来看，这事儿能成。它是全量上线的量产功能，数百项服务一次落地。在这个过程中两边的 Agent 各司其职，小布不碰账号和资金，阿宝不操作其他应用 从阶跃的介入来说，这事儿能复用。STEPX Neo 作为首个 AI 原生终端，同样的合作方式，也在通过阿宝来完成。很显然，阶跃之前并非是做硬件的，这样也能介入进来，说明方案是真的适配，手机可以，车机、眼镜也可以 再往前翻翻，去年 10 月，荣耀在 MagicOS 8 发布会上，也宣过和支付宝的智能体互通。千问 App，还有多家手机、车机、眼镜厂商，也在陆续接入 怎么看这事 对于之前的 agent 协议来说，我们还是在摘低垂的果子，只要优雅就够了 而现在，我们的对于 Agent 的落地也进入了深水区，需要能安全可落地、可闭环的标准化路径，同时解决三个问题：身份、钱、责任 而对于阿宝来说，它也给自己多定了一些验收标准： 场景可感知，就是终端得懂你在哪、要去哪 服务可执行，就是任务得真跑起来 交易可确认，就是钱的关键节点本人点头 履约可信任，就是办完的状态回得来 更重要的，是要能规模化，终端厂商不用挨个对接服务，支付宝不用挨个适配终端，一次接入，多端分发 回想一年前，大家都还把 Agent 当作 PPT 玩具，现在…我觉得不必多说，claude code/codex 一类的 agent harness 已经成为了工作中不可或缺的存在，接下来我们可能要做的，或许是让身边的各种设备接入云端，现在是 OPPO、是阶跃，再往后，想必会是车机，是眼镜…以及更多 不管哪种形式，但这肯定是整个行业对多设备智能协同的共同想象
